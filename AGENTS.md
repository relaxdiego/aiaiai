This is a **configuration-only repo**. Its sole job is wiring two AI clients
(Claude Code and pi.dev) to a LiteLLM model gateway. There is no application
code to build or test.

## What each file does

| File | Purpose |
|------|---------|
| `.envrc` | Loads devbox env; sources `.envrc.local`; derives `ANTHROPIC_BASE_URL` and `ANTHROPIC_AUTH_TOKEN` for clients |
| `.envrc.local` | **Git-ignored.** Machine-local secrets and mode. Copy from `.envrc.local.example` |
| `.envrc.local.example` | Committed template listing every required env var with placeholders |
| `devbox.json` | Provisions direnv, Python 3.12, uv |
| `litellm/config.yaml` | LiteLLM gateway config â€” model list, routing, budget. Secrets via `os.environ/VAR` |
| `pi/models.json.example` | Documents the anthropic provider override written to `~/.pi/agent/models.json` |
| `requirements.in` | **Edit this** to change the LiteLLM version (one line) |
| `requirements.txt` | **Generated** â€” full transitive lockfile with SHA-256 hashes. Regenerate with `uv pip compile requirements.in --generate-hashes -o requirements.txt` |
| `scripts/setup.sh` | Interactive wizard: asks mode, writes `.envrc.local`, installs devbox packages, writes `~/.pi/agent/models.json`, installs LiteLLM (full mode only) |
| `Makefile` | Two targets only: `make setup` and `make serve` |

## Key design decisions

**Two modes, one repo.** `MACHINE_MODE=full` means this machine runs LiteLLM
locally and the clients. `MACHINE_MODE=client` means clients only, pointing at
a gateway on another host. Only `.envrc.local` differs between machines â€” no
committed file changes.

**Single source of truth for the gateway URL.** `.envrc.local` sets
`GATEWAY_BASE_URL`. `.envrc` derives everything from it:
- `ANTHROPIC_BASE_URL=$GATEWAY_BASE_URL` â†’ picked up by Claude Code
- `ANTHROPIC_AUTH_TOKEN=$LITELLM_MASTER_KEY` â†’ picked up by Claude Code
- The setup wizard writes `$GATEWAY_BASE_URL/v1` into `~/.pi/agent/models.json` â†’ picked up by pi.dev

**No secrets in the repo, ever.** All sensitive values live in `.envrc.local`
(git-ignored). `litellm/config.yaml` references secrets as `os.environ/VAR_NAME`.
Before finishing any change, grep for leaked keys.

**Hash-locked LiteLLM.** `requirements.txt` is generated by uv with
`--generate-hashes`. Installation uses `--require-hashes` to reject any
artifact whose hash doesn't match. To upgrade: edit `requirements.in`, then
regenerate `requirements.txt`. Never edit `requirements.txt` by hand.

**pi.dev config is written, not symlinked.** The setup wizard generates
`~/.pi/agent/models.json` with the actual gateway URL substituted in. The
repo holds `pi/models.json.example` as documentation only.

## Invariants to maintain

- `requirements.txt` must always be regenerated from `requirements.in` via uv; never hand-edit it.
- `make serve` must fail clearly on a client-mode machine.
- `scripts/setup.sh` must remain idempotent: safe to re-run, warns before overwriting existing files.
- `.envrc.local` must stay git-ignored.
- No file other than `.envrc.local` should ever contain a real secret.
